# What is a Fingerprint?

A **fingerprint** is a unique identifier (hash) that Jenkins generates for a file.

Jenkins does **not** compare filenames or timestamps. Instead, it calculates an **MD5 hash** of the file contents.

For example:

```
output/result.txt
```

might generate a fingerprint similar to:

```
6dcd4ce23d88e2ee9568ba546c007c63
```

If another job later produces or consumes the **same file**, Jenkins recognizes it because the fingerprint matches.

---

# Why Fingerprinting?

Imagine a CI/CD pipeline.

```
Developer
      │
      ▼
 Build Job
      │
      ▼
 app.jar
      │
 Fingerprint
      │
      ▼
 Test Job
      │
      ▼
 Deploy Job
```

Without fingerprinting:

You don't know

* Which build created the JAR
* Which deployment used it
* Which version reached Production

With fingerprinting:

Jenkins can answer all of these.

# Real-world Example

Suppose your company builds:

```
studentportal.jar
```

Job 1

```
Compile Application
```

creates

```
studentportal.jar
```

Job 2

```
Integration Test
```

downloads

```
studentportal.jar
```

Job 3

```
Deploy Production
```

deploys the same file.

Because all jobs fingerprint the file, Jenkins can show

```
studentportal.jar

Created by:
Build #15

Used by:
Integration Test #15

Deployed by:
Deploy Production #15
```

This is extremely useful for auditing and troubleshooting.

# Where are Fingerprints Stored?

Artifacts are stored in

```
$JENKINS_HOME/jobs/<job-name>/builds/<build-number>/archive/
```

Fingerprints are stored separately in

```
$JENKINS_HOME/fingerprints/
```

Example

```
/var/lib/jenkins/fingerprints/
```

You can list them with:

```bash
ls -R /var/lib/jenkins/fingerprints
```

Each fingerprint is stored as an XML file.

# Modify the File

Run another build after changing the shell script:

```bash
mkdir -p output

echo "Student Name: John" > output/result.txt
echo "Course: Advanced Jenkins" >> output/result.txt
echo "Version 2" >> output/result.txt
```

Build again.

Because the file contents changed, Jenkins calculates a **new fingerprint** (new MD5 hash).

---

# How Jenkins Uses Fingerprints

| File Content      | Fingerprint   | Result                                 |
| ----------------- | ------------- | -------------------------------------- |
| Same content      | Same MD5      | Jenkins identifies it as the same file |
| Different content | Different MD5 | Jenkins treats it as a new file        |

# Real CI/CD Scenario

```
                Build Job
                     │
                     │
             application.jar
                     │
         Fingerprint Generated
                     │
          ┌──────────┴──────────┐
          │                     │
          ▼                     ▼
     Test Job              Security Scan
          │                     │
          └──────────┬──────────┘
                     ▼
                Deploy Job
```

At any point, Jenkins can answer:

* Which build produced this artifact?
* Which downstream jobs used it?
* Which deployment contains this exact file?
* Has this artifact changed since the previous build?

This provides traceability across your CI/CD pipeline.
